# こんどこそわかるSOLID原則

## この勉強会の目的

- 保守性の高い設計、プログラムを作れるようになること
- AIに保守性の高いプログラムを書かせられるようになること

## 保守性とは

保守性は、「変更しやすく、壊れにくいこと」を指します。
我々職業プログラマーは保守性を高めるために生きています。

- 1箇所を変更したときの影響範囲が小さい
- 変更したいときに修正箇所が少ない
- 仕様変更の見通しが立てやすい
- テストしやすい
- バグが混入しにくい
- 意図が伝わる構造・命名・分割で読みやすい

## 集約度とは

モジュール内の要素が「同じ目的に向かって集中している」度合いです。
一つのモジュールの関数・変数・データがその目的のために必要なものだけで構成されている場合高くなります。

集約度が低い例：

- 画面表示、データ取得、ログ出力が1つのクラスに混在している
- 「便利だから」と関係ない処理を同じモジュールに追加していく

集約度が高い例：

- 1つのモジュールが「注文の合計金額を計算する」など明確な役割に集中している
- 変更理由がほぼ1種類に限定される（同じ目的のためだけに変更される）

集約度が高いと理解と変更がしやすく、責務が明確になる。

## 結合度とは

モジュール同士が「どれくらい依存しているか」。低いと変更時の影響範囲が少ない。

## SOLID原則

「保守性の高い設計」のための5つの原則。

1. 単一責任の原則
2. オープン・クローズドの原則
3. リスコフの置換原則
4. インターフェース分離の原則
5. 依存性逆転の原則

この勉強会では、各原則を「なぜ必要か」「どう違反するか」「どう直すか」の順で整理する。
